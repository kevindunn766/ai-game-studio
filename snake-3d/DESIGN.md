# Snake 3D — Minimal Hyper-Casual Game

## Aesthetic
- Low-poly neon on dark background
- Snake segments: small glowing colored cubes
- Food: pulsing bright gem/cube
- Floor: subtle dark grid plane
- Minimal UI: score top-right, game-over prompt only

## Mechanics (LOCKED - do NOT add more)
1. Grid-based movement on XZ plane
2. 4-directional input: WASD or arrow keys
3. Wall = instant death
4. Eating food grows snake by 1, increases score
5. Speed increases slightly every 5 food
6. Game over screen: show score + restart prompt
7. High score persisted via config file
8. Direction input is buffered, not a single "next direction" slot (`snake.gd::_direction_queue`, capacity 2) — two quick taps thrown between grid ticks (keyboard or joystick, same code path) both get applied in order on the next two ticks instead of the first one being silently dropped. Reversal-prevention checks against the queue's tail (what direction the snake will actually be facing when that input takes effect), not the current direction, so a buffered turn can't be undone by chaining a second input onto it.

## Non-negotiable constraints
- ONE scene only: `res://scenes/Main.tscn`
- Scripts: `snake.gd`, `food.gd`, `game_manager.gd`
- Use MeshLibrary or BoxMesh for visuals — NO external art imports, EXCEPT the studio splash frames/audio (`assets/splash_frames/`, `assets/splash_audio.ogg`) and the UI font (`assets/fonts/PressStart2P-Regular.ttf`, OFL-licensed) — both explicitly user-approved, scoped exceptions to this rule, not a general opening of the door for gameplay art.
- Use StandardMaterial3D with emission for glow
- Colors must have alpha in Color() calls: `Color(r,g,b,a)`
- Godot 4.7 compatible
- Build must run headless for APK export later
- This Godot build has no Theora/video codec support compiled in (confirmed: a `.ogv` in the project silently produces no `.import` sidecar, no error) — don't reach for `VideoStreamPlayer` for anything; use an image-sequence "flipbook" (`TextureRect` + preloaded `Texture2D` array, see the splash screen) plus a separate audio track instead.

## Power-ups (LOCKED - do NOT add more types)
- Dispersed on empty non-obstacle tiles, periodic spawn check per type (never as frequent as food)
- Each of the 4 types gets its own randomly-rolled per-run spawn chance (varies run to run: sometimes rare, sometimes common)
- **Rainbow diamond** (white/rainbow gem): snake turns rainbow-colored and is immune to turret bullet shrink for 5s
- **Yellow sphere**: snake segments become shiny gold spheres, food grown gives double segments, for ~9s
- **Red diamond**: instantly spawns a burst of ~4 bonus food pieces scattered nearby, each expiring after ~8s if uncollected
- **Blue pyramid**: snake head becomes a shiny blue spike; colliding with an obstacle destroys it instead of killing the snake, for ~9s

## Title menu & leaderboards (LOCKED)

Deliberately a toggle button, not a settings screen, so it doesn't count against the "no additional menu screens" rule below.

- Two persistent leaderboards (top 5 each, saved to `user://snake3d_leaderboard.cfg`): longest time survived, and longest snake reached (peak segment count, not final length at death).
- Leaderboards are shown on both the title screen and the game-over screen; the game-over screen highlights the just-finished run's row if it placed.
- Board header 42px / row 34px font (both with a black outline for readability), 8px row spacing (`game_manager.gd::_build_board`) — sized up from the original 30/26px while keeping both boards, on both screens, comfortably within the 1080×1920 viewport with no clipping.

## Audio (LOCKED)
- All audio is procedurally generated 8-bit/chiptune waveforms via `AudioStreamGenerator` (square/pulse/triangle/noise) — no external audio files.
- `chiptune.gd` is an autoload singleton (`Chiptune`) exposing `play_sfx(name)`; any script calls it directly.

- SFX cover: eating food, high score, each power-up type, obstacle destruction, turret fire, snake taking a hit, death, and starting a run.

## NPC snakes (LOCKED)
- `enemy_snake.gd` — self-contained grid-based snake, same movement rules as the player, driven by a lightweight greedy heuristic AI (no full pathfinding — deliberate simplification for prototype scope).
- Each of 5 behaviors gets its own randomly-rolled per-run spawn chance (mirrors power-up spawn model), capped at 1 concurrent instance per behavior, first spawn no earlier than 15s into a run:
  - **Stealer** (orange): targets the nearest food that's also near the player, consumes it on contact (denies it to the player, respawns elsewhere).
  - **Boxer** (purple): heuristically intercepts a point ahead of the player's current direction rather than computing true encirclement — an approximation of "boxing in," not real pathfinding.
  - **Thief** (pink): targets the nearest active power-up, consumes it on contact (denies it to the player; does not currently grant the thief the power-up's effect, just removes it).
  - **Hoarder**: spawns with one of rainbow/yellow/blue permanently active (no timer), colored/shaped to match; otherwise pursues the player directly.
  - **Turret-head** (red, single block, never grows): pure pursuit of the player, periodically fires a homing bullet using the existing turret bullet system.
- Collision rules (mirrors classic multiplayer-snake conventions): any snake's head touching an obstacle, its own body, or another snake's body kills it. Enemies dying this way (including boxed in by the player) drop a small bonus food burst. Player death is unaffected except: while the blue spike/drill power-up is active, the player destroys any enemy snake it touches instead of dying.

## Biomes & level progression (LOCKED)
- 6 biomes = 6 levels, played in a fixed sequence within ONE continuous run (not spatial zones — `floor_manager.biome` just changes what new tiles look like as they stream in): **Neon → Desert → Glacier → Mountain → Crystal Cave → Volcanic**.
- Universal rule across all 6: floor tiles and obstacles both fill their full grid cell (no visible seam/gap) — replaced the old slightly-undersized tile meshes.
- **Neon** (original look): unchanged rainbow-hash obstacle hues, dark grid floor. Floor tiles additionally get a thin white outline (inverted-hull/cull-front rim on a shared unit mesh, `floor_manager.gd::_spawn_tile`) so the grid reads clearly — Neon-only, not applied to the other 5 biomes' floors.
- Obstacles in every biome get an outline in their own color, just brighter (`Color.lightened`) — same inverted-hull technique, per-obstacle material since the base color varies. Cleaned up alongside the obstacle itself in `destroy_obstacle_at_grid`.
- **Mountain**: grey-brown rock obstacles; floor gradient from dry grassland (far from rock) to lush green (near rock); ~18% of obstacles get a tiny procedural tree on top. The "organic Minecraft-y rock mass" look comes for free from the no-gap rule + existing neighbor-density height variance.
- **Desert**: sandy tan floor, rust-terracotta obstacles. **Glacier**: pale ice-blue floor, white-blue crystalline obstacles. **Crystal Cave**: deep purple-black floor, glowing amethyst obstacles (emission-lit). **Volcanic**: charcoal floor, dark obsidian obstacles with glowing lava-crack emission.
- **Level goal** (first time through, per level, escalating): survive continuously with at least N segments for T seconds (both N and T increase every level, `LEVEL_GOALS` in `game_manager.gd`). Dropping below N resets the sustain timer — doesn't fail the run, just resets progress toward that level's goal.
- **New Game Plus**: unlocked once after clearing level 6 for the first time (persisted forever in `user://snake3d_levelprogress.cfg`). From then on, every run's goal for each level becomes "beat your own best score recorded for that level" (per-level best scores persisted in the same file) instead of the segment/duration goal — the moment your current level's score exceeds the saved best, you advance.
- On leveling up, the biome changes for newly-streamed tiles (old tiles nearby don't retroactively repaint — creates a natural "you're entering a new area" feel as you keep moving), a "LEVEL UP" banner shows, and a small always-on HUD label (top-left) shows current level/goal progress.

## Biome-exclusive power-ups & enemies (LOCKED, additive)
- Every biome adds ONE power-up type and ONE enemy type that only spawns while that biome is active — on top of the original shared roster (all 4 original power-ups + all 5 original enemies still spawn in every biome, unchanged).
- **Neon** — power-up *Neon Bolt* (temporary 40% speed boost, ~6s); enemy *Speedster* (cyan, gets 2 moves per player step).
- **Desert** — power-up *Mirage Cloak* (enemies stop moving/reacting entirely for ~7s); enemy *Sand Burrower* (sandy, cycles hidden-underground/no-collision and surfaced-aggressive on a timer).
- **Glacier** — power-up *Ice Shield* (~4s full immunity to both bullets and enemy-collision death, small visual shield orb); enemy *Frost Wisp* (icy, only re-aims every other step — slides in a straight line between).
- **Mountain** — power-up *Boulder Burst* (instantly destroys all obstacles in a radius-2 burst around the player); enemy *Rock Golem* (grey, moves every other step but smashes through obstacles instead of dying to them).
- **Crystal Cave** — power-up *Crystal Growth* (instant +5 segments); enemy *Shard Wraith* (violet, erratic — targets the player plus a large random jitter offset, recomputed each step).
- **Volcanic** — power-up *Magma Trail* (~8s: every step leaves a glowing orange hot tile behind the player that instantly kills any enemy snake that touches it, tiles individually expire after ~4s); enemy *Magma Serpent* (glowing red-orange, direct pursuit — leaves its own trail of pink-red hot tiles as it moves, which shrink the player by 1 segment on contact instead of killing outright, same ~4s tile lifetime as the player's version).

## Mobile / Android (LOCKED)
- Portrait orientation (`window/handheld/orientation` must be the INTEGER `1`, not the string `"portrait"` — Godot silently coerces an unrecognized string to `0`/landscape with no error, which caused a real bug here once already, so don't regress this). Base viewport 1080×1920 — a standard mobile design-reference resolution (~3x scale from a typical ~360-412dp phone width baseline), `canvas_items` stretch mode with `expand` aspect so it scales cleanly across phone screen sizes.
- Movement input on touch devices is a floating/adaptive joystick (`touch_joystick.gd`) — appears wherever the player first touches the screen during gameplay, base ring 100px radius / knob 44px radius (sized per mobile game touch-target norms: primary game controls should be ~2-3x the 44-48dp minimum touch target), directions computed continuously from drag offset with an 18%-of-radius deadzone. Replaced a fixed-position D-pad, which replaced swipe-gesture detection — both prior approaches tested poorly on a real device.
- Direction is chosen by projecting the four world-grid directions through the live camera and picking whichever one's projected screen angle is closest to the touch angle (`touch_joystick.gd::_refresh_direction_angles`) — not a fixed top-down "+" cross. This makes the stick's up/down/left/right correspond to the fixed isometric camera's actual rotated view instead of the raw world axes. Recomputed fresh at the start of every touch (never cached across a gesture or a restart), so it can't drift or go stale over multiple playthroughs. The currently-held direction gets an extra ~14 degrees of hysteresis so small thumb wobble near a boundary doesn't flicker between two directions — a real-device fix for the stick feeling like it needed a precise, "stretched" push to register a turn.
- The joystick node always exists once built; visibility is purely a function of active touch (shown in `begin_touch`, hidden in `end_touch`) rather than a separate "layer visible" flag toggled only at first game start — this was a deliberate fix for a bug where controls stopped appearing after the first restart (the flag never got reset on the `skip_menu_on_ready` restart path). Don't reintroduce a start-only visibility flag for any future touch control.
- Tap-anywhere-on-screen still starts the game / restarts after death, independent of the joystick.
- Leaderboards are stacked vertically (not side-by-side) on both the title and game-over screens — appropriate for a narrow portrait canvas, unlike the original landscape-era layout.
- Package name: `com.hellcoilstudio.snake3d`; app display label is "Battle Boa" (`package/name` in `export_presets.cfg`). Android SDK + export templates are set up locally — see the `env-quirks` memory for exact paths if re-setting-up on a fresh machine.
- Haptic feedback (`Input.vibrate_handheld`, no-op on desktop so safe to call unconditionally) on power-up pickup, high score, level up, taking a hit, and death (`game_manager.gd::_haptic`). Deliberately NOT on eating food — too frequent an event, would read as constant buzzing rather than meaningful feedback.
- Title-screen showcase mode: while on the title/menu screen (and during the splash before it), `floor_manager.gd` builds a much larger radius of tiles than gameplay ever needs (`MENU_SHOWCASE_RADIUS`, an estimate — the menu camera's orthogonal size is 14x gameplay's, so the normal small streaming radius left visible gaps) and splits it into 6 angular pie-slice wedges around the spawn point, one per biome (`_effective_biome_for`), rather than showing only the current level's single biome — a deliberate exception to the normal streaming-radius optimization, scoped to the menu only. Reverts to normal per-segment-count radius and the run's actual single biome the instant `_start_game()` fires; the existing despawn-tween culling shrinks the excess away over the following frames. Tile pop-in tweens are skipped for showcase tiles specifically (thousands built in one frame — a tween burst at that scale was judged not worth the risk untested).

## Camera (LOCKED)
- Orthographic crane camera size scales with segment count (`seg_count + 4`, clamped 5-40) so the view zooms out as the snake grows.
- Additionally nudges out (up to +6, clamped overall to 46) when an obstacle is within a 5-tile scan radius of the head — proportional to how close the nearest one is. Keeps early-run visibility reasonable when the snake is short (and therefore zoomed in tight) but a hazard is already close by.
- Tilt-shift photography effect (`shaders/tilt_shift.gdshader`, full-screen overlay `ColorRect` on `TiltShiftLayer`): a horizontal focus band stays sharp, everything above/below blurs via mip sampling. Gameplay uses `TILT_SHIFT_NORMAL_FOCUS_WIDTH`/`_MAX_BLUR`, the title screen a stronger `TILT_SHIFT_MENU_*` pair, swapped in `_apply_gameplay_camera_state`/`_apply_menu_camera_state`. Strengthened from the original values (narrower focus band, higher max blur; shader's `max_blur` hint_range ceiling raised 5→8 to give the menu value headroom) for a more pronounced miniature-diorama look.

## Player weapons (LOCKED)
- Every current snake segment mounts a small turret (metallic base + glowing green tip, `snake.gd::_make_turret_node`) — hidden on the head while the blue spike power-up is active so it doesn't clash visually.
- Every `PLAYER_TURRET_FIRE_INTERVAL` (2s), each segment independently checks for the nearest living enemy snake within `PLAYER_BULLET_RANGE` (14 tiles) and fires a homing bullet at it if one exists — no target, no shot from that segment that cycle. More segments kept alive means more simultaneous bullets per volley; losing segments loses firepower, not fire rate.
- A player bullet that reaches its target kills that enemy (same `_kill_enemy` path as other enemy deaths, including the bonus food drop); a bullet whose target dies or leaves range first re-targets the next-nearest enemy, or fades out if none remain.
- Visually and mechanically separate from the existing environment/enemy turret bullets (which still hunt the player) — distinct green color, separate `_player_bullets` list.
- Both bullet lists are ticked unconditionally at the very top of `_process`, before the `game_state`/`is_dying` early returns — a bullet in flight keeps homing, popping, and expiring through the death sequence and the game-over screen instead of hanging frozen in place. Don't move bullet ticking back below those checks. The player-hit side effect (`_shrink_snake`) still only applies while `game_state == PLAYING and not is_dying`, so a bullet finishing its flight after death doesn't double-trigger death handling.
- Every bullet exit path — hit, timed out, or (player bullets only) lost its target with nobody left to re-target — goes through `_destroy_bullet_on_impact`: hide + detach from tree + `queue_free`, synchronously, same frame, no tween-and-callback anywhere in bullet cleanup. Also has a same-frame "didn't actually move" backstop after every position update, as a second line of defense.
- Root causes of the "bullets hit and just pile up" bug — confirmed live via `adb logcat` while driving the game with simulated input, not guessed. Three separate real bugs, all now fixed:
  1. When several bullets in the same volley targeted the same enemy, the one that landed first killed it, but `queue_free()` defers actual deallocation — the *other* bullets still homing on it saw `is_instance_valid(target) == true` for another frame or two and kept chasing its last (frozen) position. Fixed by also checking `enemy_snakes.has(target)` before trusting a cached target — `enemy_snakes.erase()` in `_kill_enemy` is synchronous, unlike the deferred free.
  2. `chiptune.gd::play_sfx` called a nonexistent `AudioStreamGeneratorPlayback.clear()` — the real method is `clear_buffer()`. This silently errored on nearly every sound effect in the game (pre-existing, not introduced by any of the above). Low-impact on its own (that one sfx just didn't play), but worth knowing this was broken for anyone touching audio.
  3. The real jam: both bullet-tick loops cast `_b as MeshInstance3D` *before* checking validity. Casting an already fully-deallocated (not just queue_free-pending) object throws "Trying to cast a freed object", and separately, `var target: Node3D = b.get_meta(...)` on a fully-freed stored reference throws "Trying to assign invalid previously freed instance" — both **before** the following `is_instance_valid()` check ever runs. Either error aborts the whole loop immediately, skipping the `to_remove` cleanup at the end entirely, which permanently jams every bullet behind the bad entry in the array — new bullets keep getting added on top forever after. Fixed by checking `is_instance_valid()` on the untyped reference *before* any cast or typed assignment, in both `_tick_bullets` and `_tick_player_bullets`. This is the pattern to follow for any future bullet-like array: validity check first, on the raw untyped value, always.

## Bonus power-ups (LOCKED, additive)
- Three more global power-ups (not biome-exclusive, spawn in every biome alongside the original 4), each with its own randomly-rolled per-run spawn chance like the rest:
  - **Laser** (cyan rod): for 5s, a continuous beam fires from the head along the current movement direction, stopping at and destroying the first obstacle in its path, and killing any enemy snake segment caught along its current length. Re-evaluated ~7x/sec (`LASER_TICK_INTERVAL`) so the beam visual and its effect stay in sync as the snake moves/turns. A continuous particle trail is a child of the beam mesh (inherits its position/length automatically) so the beam reads clearly even when it isn't currently touching anything.

Always spawns a particle burst at the head on activation regardless of whether any enemies were actually in range, so the power-up is never silently a no-op when the run happens to be enemy-free at that moment.
  - **Nova** (orange sphere): instant. Destroys every obstacle and kills every enemy snake within `NOVA_RADIUS` (4 tiles) of the player — the bigger, non-biome-locked, "everything" version of Boulder Burst.

## Dynamic difficulty (LOCKED, additive)
- The target audience is children, so the game rubber-bands to the player's recent success across runs. A single persisted dial `difficulty` (0 = easiest, 1 = hardest, saved in `user://snake3d_difficulty.cfg` alongside a rolling `history` of recent-run performance) drives every knob below (`game_manager.gd::_apply_difficulty`). Resolved once per run in `_ready()` (a restart reloads the whole scene), so the dial only shifts between runs, never mid-run.
- **Adaptation** (`_update_difficulty`, called once at death with the finished run's score + survival time): a run's performance = `score + play_time * DDA_TIME_WEIGHT`, compared against the average of the last `DDA_HISTORY_SIZE` (5) runs. Below the baseline by more than a ±15% dead zone → ease one `DDA_STEP` (0.12) toward the easy floor; above it → harden one step toward the hard ceiling; inside the band → unchanged. So a steady decline walks the dial to the floor and a steady climb walks it to the ceiling (the "respective min/max"), while noise/one-off runs don't twitch it. Default starts at `DDA_DEFAULT` 0.4 (slightly below neutral).
- **Knobs** each lerp between an EASY endpoint (dial 0) and a HARD endpoint (dial 1); the clamp to [0,1] is the min/max. Easing boosts what HELPS the player and softens what THREATENS it: more food kept on the board (`_get_target_food_count` food bonus), higher power-up spawn chance + shorter power-up check interval, slower environment-turret volleys (`_dda_env_turret_interval`) and slower enemy turret-head fire (`EnemySnake.fire_interval_scale`, set at spawn), lower enemy spawn chance, longer enemy spawn-check interval, and a later first-enemy grace period. Hardening does the reverse.
- Deliberately does NOT touch the player's own per-segment turrets (`PLAYER_TURRET_FIRE_INTERVAL`) — slowing those would make an "easier" run harder, so they stay fixed. No new spawn/enemy/power-up *types* are introduced; this only scales the rates/counts of the existing roster.

## Transition curtain (LOCKED, additive)
- A loading-screen wipe that carries the splash's decorative-snake idea into gameplay, but in the gameplay **isometric** view (yaw 45 / pitch -35.264, orthographic — matches the gameplay crane) instead of the splash's flat top-down lane. Several snakes (`TRANSITION_SNAKE_LANES`) crawl **diagonally** across the whole screen; even lanes travel one way, odd lanes the other, so directions **alternate** lane by lane. Rendered in an isolated `SubViewport` (own World3D, opaque black bg — same setup as the splash decor snakes), shown full-screen on top of a black curtain whose opacity fades in then out. This is NOT a menu screen (no buttons, no state the player navigates) — it's a timed effect, so it doesn't count against the "no additional menu screens" rule.
- Shown at exactly two moments: **menu → gameplay start** (`_begin_menu_to_game_transition`) and **between playthroughs on restart** (`restart`). For its whole lifetime it **refuses all input** (`is_transitioning` gates the top of `_unhandled_input`) and **freezes the run** (`is_transitioning` gates the top of `_process`, so the snake behind the curtain doesn't step and the timer doesn't advance until the reveal finishes — the new run can't move before the player can control it).
- Driven analytically from `_transition_time` in `_tick_transition` (no tweens) as **three strictly sequential beats** so no visible animation ever overlaps the heavy load frame: **COVER** fades black 0→1 with the snakes parked off-screen (nothing sweeps across the fade); the level is loaded/swapped at the fully-black moment; **RUN** holds the curtain opaque and sweeps the snakes all the way across (progress 0→1) as the loading beat; **REVEAL** fades black 1→0 once they've crossed off-screen. A restart reloads the whole scene, so its transition is **split across the reload**: COVER runs in the old scene and triggers `reload_current_scene` at full black; the reloaded scene's `_ready()` opens already-black (`_play_run_on_ready`, a `static var` so it survives the reload) and plays RUN + REVEAL. The menu→gameplay case has no reload, so it runs COVER → hidden `_start_game()` swap → RUN → REVEAL in one scene. The snake sweep always runs as its own beat over full black (after the load, before the reveal), never during a fade — the deliberate ordering that keeps the sweep smooth regardless of how heavy the load frame is.
- The curtain snakes are **lit** (directional key light + ambient in the SubViewport), not unshaded like the splash decor, so each cube face reads as a distinct tone; `_make_transition_snake` builds them scaled up from the splash size.
- Visual proportions (`TRANSITION_CAM_SIZE`, `TRANSITION_SNAKE_SPAN`, `TRANSITION_SNAKE_LANE_SPACING`, lane count, `TRANSITION_FADE_TIME` for the fades / `TRANSITION_RUN_TIME` for the snake-sweep beat, colors, light angle/energy) are tunable constants at the top of the transition section.
- **Stutter avoidance (don't regress):** the transition itself is cheap; the heavy work it hides is the scene reload (restart) and the floor's initial fill. Two things keep it smooth: (1) `_tick_transition` clamps its per-frame step so a heavy frame can't teleport the animation, and (2) `floor_manager._update_tiles` builds the initial **bulk fill instantly** (no per-tile pop-in tween) — hundreds of simultaneous pop-in tweens firing as the curtain revealed was the "loading screen" stutter (tiles visibly scaling in during the fade). Pop-in tweens are kept only for the handful of tiles that stream in per-step during real play.

## Game-over hold (LOCKED)
- When a run ends, the leaderboard/recap holds for `GAME_OVER_TAP_LOCKOUT_MS` (3s) during which all restart inputs (tap / R / Enter) are ignored, so a stray tap can't skip past the result; the "[ RESET ]" prompt stays hidden until the lockout lifts as the cue that the next tap advances (into the restart transition). Mid-run R-restart is unaffected (`_game_over_input_locked` is only true once a run has ended).
- The game-over recap card (segments / time / enemies / biome) is laid out with **plain anchored labels positioned by hand, NOT a VBoxContainer** — `_show_recap` animates each line's position for the slide-in, and a container re-sorts its children every frame and overrode those positions, piling all four lines on top of each other (the overlap bug). Don't put them back in a container.
